El recorrido

Qué pasa en una pregunta

1
Usuario pregunta: "¿cuántos PDFs hay y cuál es el más grande?"
2
El modelo local decide llamar una herramienta y emite un tool_call.
3
Se ejecuta folder_stats() — recorre la carpeta dentro del sandbox y devuelve conteos exactos.
4
El resultado vuelve al modelo, que redacta la respuesta en lenguaje natural.
5
El checkpointer guarda el turno: la siguiente pregunta ("¿y de esos…?") ya tiene contexto.
Cómo se desplegó el endpoint

El modelo, servido con Docker

Imagen oficial de llama.cpp con CUDA, fijada a la RTX 5070. Expone un endpoint OpenAI-compatible que ambos stacks consumen.

docker/01_serve_model.sh
# Qwen2.5-7B con tool-calling, sobre GPU, en :8081
docker run -d --rm --name demo-llm \
  --gpus '"device=0"' # pin a la RTX 5070 (12GB)
  -p 8081:8081 \
  -v $HOME/models/qwen2.5-7b-instruct:/models \
  ghcr.io/ggml-org/llama.cpp:server-cuda13 \
  -m /models/Qwen2.5-7B-Instruct-Q4_K_M.gguf \
  --jinja # ← imprescindible: habilita tool-calling
  -c 32768 # contexto 32K
  -ngl 99 --flash-attn on \
  --host 0.0.0.0 --port 8081
Lo que lo hace robusto

Memoria y seguridad

🧠 Memoria de conversación

  • LangGraph: checkpointer + thread_id por sesión.
  • Compactador: recorta a las últimas N intervenciones sin romper la secuencia tool_call/result, conservando el system prompt.
  • Agent SDK: memoria y compactación de fábrica (el harness las gestiona).

🛡️ Seguridad (carpeta real)

  • Sandbox: bloquea path traversal, rutas absolutas y symlinks que escapan.
  • Solo lectura por defecto: move_file no hace nada sin permiso explícito.
  • Las tools nunca crashean: degradan a un mensaje de error.
